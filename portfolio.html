Marazon

Game Link

Description

Marazon is a game where you use the arrow keys to move the character
and try to reach the goal line. While playing the game, the player can
collect apples to get more points but they must be aware that touching
the enemy sprite will cause them to lose the game. The apples will be
spawned randomly and so will the enemy when it first appears. The larger
the size of your character, the higher the difficulty is.

Reflection

This game was created with the goal of reaching the goal while being
blocked by obstacles. Inspiration for the game came from other games
that had the player use the character and try to dash to the end of the
stage. In the development process, two successsful points were that the
sprites appeared or disappeared on cue and that the player could choose
the character that he/she would use. Two obstacles were getting the game
to play the next stage and creating apples. In the game, the player is
easily able to transition to the stage but must be aware that some apples
are fakes aimed with delaying the player. If there was more time, aspects
of the game that would be improved would most likely adjusting how the
difficulty works and maybe making it so that the apples are individually
collected instead of a mass at once.

More Blocks + Algorithm

An algorithm gives the well-defined procedure of how the computer should
solve a problem. This algortihm is also a "More Blocks" and is defined to
tell the computer how to handle the sprite and do certain things when the
conditional is fulfilled. In particular, it changes the color effect by an
integer five times and then plays a finger snap.

Bank Heist

The user is at a bank. Through the user's decisions, the user can choose
to be a security guard or a bank goer. The paths diverge so the user can choose
to attempt to stop a robbery, stop a robbery, do nothing about the robbery,
or even commit the robbery.

My partner and I were assigned to create a story that with user input
could arive at multiple endings. The first and largest problem was what to
make the story about. We brainstormed with random ideas and examples that
we previously knew. That was when my partner thought of an idea we agreed
on, a bank setting with a robber. As we began our story on Google Drawings,
we began to make more decision points and routes that the user could follow
until we ended up diverging from our original plan of having the robber
route as the main path. After using repl.it to implement a Python program,
we started off with importing code from our teacher that would type out
lines of text instead of printing the block of text all at once. Then, it
was finally time to define the scenes of the story. A defined scene would
ask the user to accurately input a plot choice and would then use if-else
statements to call certain scenes. For example, the user is told that they
are at a bank and are asked to input whether they want to be a security
guard or a bank goer. Depending on the user's choice, they would be sent
down the guardLife() or the personLife(). The process of defining scenes,
asking for user input, and writing if-else statments was repetive as we
followed the flow chart we made about how users could diverge or converge
their paths. Since we were able to finish coding the story early, I decided
to go back to our original plan of a robber and made that a hidden route
in the diverging paths for a bank goer.

The difficult part came when we had to debug the program, making sure
that each pathway worked properly and that the grammar was on point. As we
were constantly running the program, a problem came up, the moderate typing
speed would be troublesome as we needed to run the program multiple times.
Therefore, we used a shortkey to find and replace all the numbers into a
variable. Lastly, we had to make sure that all the code involving user input
would work. This included while loops to confirm that the program would print
out 'Invalid' if the user input was not in the choices provided and fixing
any writing typos in if-else statements.

Organizing scenes of our story into functions is a form of procedural
abstaction because it assigns each scene a function and gives it a certain
task of leading the user to the next scene which is very simple. Instead of
following the option of making continuous nested if-else statements, we called
and defined scenes into new functions because it gives the advantages of
reducing complexity and allows for implementation of more complex procedures.
